\## 🧭 The Big Picture

Your phone shows full bars. But you still can\'t load a page. The call
drops the moment you walk into a corridor.

This is not a network outage --- it is \*\*physics\*\*. Reinforced
concrete, metal window frames, and dense building layouts act as RF
shields, creating invisible cellular dead zones that no amount of
carrier infrastructure can fix from the outside.

\*\*Project Echelon\*\* set out to prove this empirically. We walked the
entire SIT Pune campus with our phones acting as scientific instruments
--- logging GPS coordinates, signal strength, and cell tower data every
few seconds --- and then built a complete data pipeline to process,
analyse, and spatially visualise that data.

The result is a two-part system: 1. A \*\*Jupyter Notebook\*\*
(\`Project_Echlon.ipynb\`) that cleans the raw data and performs
statistical Exploratory Data Analysis (EDA) --- proving dead zones
mathematically through graphs and distributions. 2. A \*\*Python
script\*\* (\`check_10.py\`) that deploys the same data as a live,
interactive satellite heatmap --- proving dead zones \*spatially\*,
showing exactly where on campus your signal collapses and why.

\## 📱 Phase 1 --- Real-World Data Collection

This is where the project begins --- not on a laptop, but on a walk.

\### How We Collected the Data

We used our personal Android phones to log cellular RF telemetry across
the SIT campus. The primary metric we captured is \*\*RSRP --- Reference
Signal Received Power\*\* --- the standard telecom industry measurement
of LTE/5G signal strength, expressed in negative decibels (dBm). A
reading of \`-65 dBm\` is excellent. Anything past \`-105 dBm\` is
effectively a dead zone where calls drop and data stalls.

The apps used were configured to log data continuously as we walked
predetermined paths covering: - The main academic block - Corridors and
building interiors - The hostel zone - The campus perimeter and main
gate road

\### What the Apps Logged (per GPS ping)

\| Field \| Description \| \|\-\--\|\-\--\| \| \`lat\` / \`lon\` \| GPS
coordinates at the moment of the reading \| \| \`signal\` \| RSRP in dBm
--- the actual signal strength value \| \| \`mnc\` \| Mobile Network
Code --- a number identifying which ISP tower is serving the phone \| \|
\`cellid\` \| Unique identifier of the specific cell tower currently
connected \| \| \`time\` \| Unix millisecond timestamp of the reading \|

Each walk across the campus generated hundreds to thousands of data
points. Combined across all team members and all ISPs (Jio, Airtel,
Vodafone), we ended up with \*\*1,363 total telemetry data points\*\*
--- a statistically meaningful sample of the campus RF environment.

\### Why Multiple ISPs?

Different carriers use different tower placements and frequency bands.
By collecting data simultaneously for Jio, Airtel, and Vodafone, we
could compare not just \*where\* coverage is weak, but \*which ISP\*
performs better in each zone --- turning a simple coverage map into a
competitive network audit.

\## 🧹 Phase 2 --- Data Ingestion & Sanitization (Notebook)

\*\*File:\*\* \`Project_Echlon.ipynb\` \| \*\*Platform:\*\* Google Colab

Raw hardware telemetry is messy. Sensors miss readings. GPS locks slip.
Apps from different manufacturers export columns with different names.
Phase 2 is a rigorous, 7-step sanitization pipeline that transforms
noisy, multi-source CSV files into a clean, mathematically valid
baseline dataset.

\### The 7-Step Pipeline

\*\*Step 1 --- Data Ingestion\*\* All uploaded CSV files are discovered
automatically using \`glob\` and concatenated into a single master
\`DataFrame\`. No manual file naming required.

\*\*Step 2 --- Format Standardization\*\* Column headers are normalized
to lowercase across the entire \`DataFrame\`. This prevents key errors
caused by different apps exporting \`Lat\`, \`LAT\`, or \`latitude\` for
the same field.

\*\*Step 3 --- Feature Selection\*\* The dataset is stripped down to
only the 7 columns needed for RF spatial analysis: \`lat\`, \`lon\`,
\`signal\`, \`mnc\`, \`cellid\`, \`alt\`, \`time\`. All other columns
(speed, accuracy, bearing) are discarded.

\*\*Step 4 --- Null Value Sanitization\*\* Any row with a missing GPS
coordinate or signal reading is dropped. Passing a \`NaN\` coordinate to
the mapping engine causes a fatal error, so this step is non-negotiable.

\*\*Step 5 --- Duplicate Filtering\*\* When a phone is stationary, it
logs the same coordinates multiple times per second. These duplicate
location-timestamp pairs are dropped to prevent artificial inflation of
signal density in the heatmap.

\*\*Step 6 --- Physical Limits Validation\*\* Cellular RSRP is always a
negative number. A positive signal value indicates a hardware glitch.
This step enforces that all signal values are negative and all
coordinates are non-zero and physically valid (not \`0,0\` --- the Gulf
of Guinea).

\*\*Step 7 --- Type Casting & Time Conversion\*\* All columns are cast
to strict numerical types (\`float\` for coordinates, \`int\` for signal
and MNC). Unix millisecond timestamps are converted to human-readable
IST \`DateTime\` objects using Pandas, enabling chronological analysis.

\### Output of Phase 2

\`\`\` Echelon_Cleaned_Telemetry.csv \`\`\` A sanitized, validated,
typed baseline dataset --- ready for analysis and mapping without
re-running the cleaning pipeline.

\## 📊 Phase 3 --- Exploratory Data Analysis & Graphing (Notebook)

\*\*File:\*\* \`Project_Echlon.ipynb\`

With clean data in hand, Phase 3 moves from engineering to analysis.
Before deploying any map, we first need to understand the \*mathematical
structure\* of the campus RF environment. This phase engineers new
features from raw data and generates four key visualizations.

\### Feature Engineering

\*\*MNC → ISP Name Mapping\*\* The raw \`mnc\` integer codes are
translated into human-readable provider labels using a custom mapping
function:

\| ISP \| MNC Codes \| \|\-\--\|\-\--\| \| Jio \| 854--874 \| \| Airtel
\| 10, 31, 40, 44, 45, 49, 51--56, 70, 84, 90--98 \| \| Vodafone (Vi) \|
20, 27, 43, 46, 60, 88 \| \| BSNL \| All others (fallback) \|

\*\*RSRP → Signal Quality Categorization\*\* Raw dBm values are bucketed
into four telecom-standard quality tiers:

\| Category \| RSRP Threshold \| Meaning \| \|\-\--\|\-\--\|\-\--\| \|
🟢 Excellent \| ≥ -75 dBm \| Strong, reliable connection \| \| 🟡 Good
\| -76 to -90 dBm \| Normal everyday usability \| \| 🟠 Fair \| -91 to
-105 dBm \| Slow data, occasional buffering \| \| 🔴 Dead Zone
(Critical) \| \< -105 dBm \| Call drops, no data throughput \|

\-\--

\### The Four Graphs

\*\*Graph 1 --- Campus RF Signal Distribution (Histogram)\*\*

A histogram of all RSRP values across the campus, overlaid with a KDE
(Kernel Density Estimate) curve. A vertical dashed line marks the -105
dBm dead zone threshold.

\*What to look for:\* If the distribution has a statistically
significant tail extending past the dashed line, dead zones are
mathematically proven to exist --- not just anecdotally reported.

\-\--

\*\*Graph 2 --- Signal Quality Classification Count (Bar Chart)\*\*

A bar chart counting how many data points fall into each of the four
quality tiers (Excellent / Good / Fair / Dead Zone), colour-coded green
to red.

\*What to look for:\* The ratio of Dead Zone bars to Excellent bars
quantifies exactly what percentage of the campus has reliable
connectivity vs. what percentage is failing.

\-\--

\*\*Graph 3 --- Top 10 Most Active Cell Towers (Bar Chart)\*\*

A ranked bar chart of the 10 Cell IDs with the highest number of pings
during the campus survey.

\*What to look for:\* A highly skewed distribution (one or two towers
with far more pings than the rest) indicates the campus is being served
by a small number of overloaded macro cells --- a infrastructure
bottleneck that explains poor performance even in areas with technically
acceptable signal.

\-\--

\*\*Graph 4 --- The \"Walk of Frustration\" Signal Volatility (Line
Plot)\*\*

A time-series line plot of RSRP values across the first 150 sequential
GPS pings --- representing a continuous walk across campus. The region
below -105 dBm is shaded red.

\*What to look for:\* Sharp drops into the red zone correspond to the
exact moments a person walking that path would experience a call drop or
data stall. This graph makes the invisible experience of bad
connectivity viscerally visible.

\## 🗺️ Phase 4 --- Geospatial Deployment (Mapper Script)

\*\*File:\*\* \`mapper/check_10.py\`

The final phase takes the same cleaned data and renders it spatially ---
on a real satellite image of the campus. Where Phase 3 proved dead zones
\*statistically\*, Phase 4 proves them \*geographically\*, answering the
question: \*exactly which building, which corridor, which corner of
campus\* is the problem?

\### How the Script Works

\*\*Data Ingestion (CSV + KML)\*\* The script auto-discovers all
\`.csv\` and \`.kml\` files in the working directory. CSVs are read with
Pandas; KML files (Vodafone route data) are parsed with BeautifulSoup,
extracting signal values from embedded \`{value}dBm\` name tags.

\*\*ISP Routing\*\* Each data point is classified by MNC into one of
four intercept lists (Jio, Airtel, Vodafone, Unknown) before any
rendering begins. This allows each ISP to have its own independent,
toggleable map layer.

\*\*Satellite Basemap\*\* The map is built on real \*\*Esri World
Imagery\*\* satellite tiles --- the same satellite data used by ArcGIS
and Apple Maps --- giving the heatmap geographic context that a generic
street map cannot provide. Every data point lands on the actual
building, road, or open area it was recorded at.

\*\*The Dead Zone Override Protocol\*\* This is the most critical
technical decision in the mapping pipeline. A standard heatmap gradient
blends all signal strengths together --- which means a cluster of -110
dBm readings (critical failures) gets visually absorbed into the
surrounding orange-yellow gradient and becomes invisible.

To prevent this, the script separates the rendering logic: any point
below -105 dBm is \*not\* added to the heatmap. Instead, it is rendered
as a \*\*hard red \`CircleMarker\`\*\* on its own dedicated
\`FeatureGroup\` layer (\`⚠️ \[ISP\] CRITICAL DEAD ZONES\`). These
markers sit on top of everything else and cannot be washed out by
surrounding signal data.

\*\*Layer Architecture\*\* Each ISP gets three independent, toggleable
layers:

\| Layer \| Default \| Description \| \|\-\--\|\-\--\|\-\--\| \|
\`\[ISP\] Heatmap\` \| ✅ ON \| Smooth density gradient (green → yellow
→ orange) \| \| \`\[ISP\] Raw Data Nodes\` \| ☐ OFF \| Individual
precision dots at each GPS ping \| \| \`⚠️ \[ISP\] CRITICAL DEAD ZONES\`
\| ✅ ON \| Hard red circles at all \< -105 dBm locations \|

\*\*Building Labels\*\* Google Maps-style white text labels with dark
drop-shadows float permanently over the satellite imagery, identifying
key campus landmarks (\`SIT Pune Campus Area\`, \`SIT Boys/Girls
Hostel\`, \`Main Gate / Road\`). These are fully customizable in the
\`BUILDING_LABELS\` dictionary at the top of the script.

\*\*Dashboard Panel\*\* A fixed bottom-left panel displays the total
data point count and a visual signal quality legend --- giving the map
the feel of a professional network operations dashboard rather than a
student project.

\## 📊 Key Findings

From the 1,363 data points captured across SIT Pune campus:

\- \*\*Dead zones are real and structurally caused.\*\* The RSRP
histogram shows a statistically significant tail past -105 dBm,
concentrated in building interiors and enclosed corridors --- confirming
that campus architecture attenuates RF signals in predictable patterns.

\- \*\*Coverage is ISP-dependent and spatially uneven.\*\* Jio shows the
broadest campus coverage. Vodafone signal is considerably sparser,
particularly in the hostel zone. No single ISP provides uniform coverage
across the entire campus.

\- \*\*Cell tower load is heavily concentrated.\*\* The tower hopping
graph reveals that the campus is predominantly served by a small number
of macro cells, creating handover overhead and load bottlenecks during
peak usage hours.

\- \*\*The main gate road zone has the best signal across all ISPs.\*\*
Open-air, away from building obstructions, and close to road-facing
tower infrastructure --- this aligns with standard RF propagation
theory.

\- \*\*Building interiors are the primary failure zone.\*\* The
geospatial map confirms that virtually all hard dead-zone markers (red
circles) cluster around building interiors, stairwells, and ground-floor
zones below floor level --- consistent with the physics of RF signal
attenuation through reinforced concrete.

\## 📌 Limitations & Future Scope

\*\*Current Limitations\*\* - Data represents a single survey session;
multi-day, multi-path collection would improve statistical confidence -
The pipeline does not distinguish indoor from outdoor readings --- a key
variable for RF attenuation analysis - Building labels are manually
geocoded; a production version would derive them from OpenStreetMap
building polygons - The mapper script has no CLI interface --- folder
path, zoom level, and output filename are all hardcoded

\*\*Future Extensions\*\* - \*\*5G NR Metrics:\*\* Add RSRQ and SINR
columns for a more complete RF picture beyond RSRP alone - \*\*Temporal
Analysis:\*\* With enough data across multiple days, plot signal
strength vs. time-of-day to detect congestion patterns during peak class
hours - \*\*Predictive Dead Zone Modelling:\*\* Train a spatial ML model
on the cleaned data to predict dead zones purely from building layout
geometry --- without needing field measurement - \*\*CLI Interface:\*\*
Add \`argparse\` support so the mapper can be run with \`\--folder\`,
\`\--zoom\`, and \`\--output\` flags
